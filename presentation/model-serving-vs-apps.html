API Contract

Responses API Format

The SAME request/response format for both Model Serving and Apps

Request ResponsesAgentRequest

# Standard invoke request { "input": [ {"role": "user", "content": "What products match 'drill'?"} ] } # Streaming request — add stream: true { "input": [ {"role": "user", "content": "What products match 'drill'?"} ], "stream": true } # Multi-turn: include prior assistant turns { "input": [ {"role": "user", "content": "Find drills"}, {"type": "message", "role": "assistant", "content": [{"type": "output_text", ...}]}, {"role": "user", "content": "How much?"} ] }

Response ResponsesAgentResponse

# Invoke response — output array { "output": [ { "type": "message", "role": "assistant", "content": [ {"type": "output_text", "text": "Found 3 drills..."} ] } ] } # Stream response — SSE events # event: response.output_item.added # data: {"type": "message", ...} # event: response.output_text.delta # data: {"delta": "Found 3..."} # event: response.output_item.done # data: {"type": "message", ...} # event: response.completed

Both /serving-endpoints/{ep}/invocations and /apps/{app}/invocations accept this identical format

Security

Auth: Model Serving

SystemAuthPolicy (auto SP) + UserAuthPolicy (OBO opt-in)

  • SystemAuthPolicy: Auto-provisioned SP via resources param in log_model()
  • SP gets "Can Query" on each declared serving endpoint
  • UserAuthPolicy: OBO opt-in via on_behalf_of_user=True per resource
  • Uses ModelServingUserCredentials to obtain user token at runtime
  • Dual policies work together: system resources use SP, user-facing resources use OBO
  • Default: all requests run as the SP identity unless OBO is explicitly enabled
# OBO opt-in per resource resources=[ DatabricksServingEndpoint( endpoint_name="llm-endpoint"), DatabricksServingEndpoint( endpoint_name="user-endpoint", on_behalf_of_user=True), # OBO ]

Security

Auth: Databricks Apps

X-Forwarded-Access-Token + OAuth consent + user_api_scopes

  • Platform injects X-Forwarded-Access-Token header per request
  • get_request_headers() from AgentServer captures the token
  • Token creates WorkspaceClient(token=forwarded_token)
  • Users see OAuth consent screen on first visit
  • Dual auth: OBO for user resources, App SP for system resources

Observability

MLflow Tracing

Model Serving AUTOMATIC

Infrastructure handles everything. Traces appear in the linked MLflow experiment automatically.

# In src/agent.py — that's it! import mlflow mlflow.langchain.autolog() # ResponsesAgent.predict_stream() # auto-traces all LangGraph spans # to the serving endpoint's experiment
Databricks Apps MANUAL SETUP

You must explicitly configure the tracking URI, experiment, and autolog in app startup.

# In app/main.py — before AgentServer import mlflow # Required: point to workspace MLflow mlflow.set_tracking_uri("databricks") mlflow.set_experiment( "/Workspace/Shared/my-app" ) mlflow.langchain.autolog() # Then define @invoke, @stream, AgentServer

UC-Based Trace Storage (Both Paths)

Decision Guide

When to Use Which

☁ Choose Model Serving

  • Standard chat agents with OpenAI-compatible API
  • Production workloads requiring autoscaling
  • When you want fully managed infrastructure
  • Need A/B testing and traffic splitting
  • Review App for feedback collection
  • Quick POC with minimal code

📦 Choose Databricks Apps

  • Custom UIs beyond a chat interface
  • Complex multi-step workflows with custom logic
  • Internal tools with per-user authorization (OBO)
  • Custom endpoints (extend AgentServer's FastAPI app)
  • Rapid iteration during development
  • Full-stack apps (frontend + backend)

Pro tip: Start with Model Serving for your MVP, then migrate to Apps when you need customization. Or use both: Model Serving as the agent backend, Apps as the custom frontend.

Things to Know

Additional Considerations

🔄 A/B Testing

Model Serving supports native traffic splitting between model versions. Apps require a custom routing layer.

🏭 Framework Flexibility

AgentServer builds on FastAPI — extend its app object with custom routes, or use Streamlit/Gradio alongside.

🏛 Unity Catalog Governance

Both paths integrate with UC. Model Serving uses UC model registry; Apps use UC for data access via OBO.

🛡 Rate Limiting

Model Serving has built-in rate limiting. Apps need custom middleware (e.g., slowapi).

🌐 Static Assets

Apps can serve HTML, CSS, JS, images — build rich interactive UIs alongside your agent.

🔀 Hybrid Approach

Use Model Serving as the agent backend and an App as the custom frontend — best of both worlds.
